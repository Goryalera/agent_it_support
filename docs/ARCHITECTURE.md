# AskOps — архитектура

> V1 — фундамент (оркестратор + RAG-агент). **Слой 1** добавил аналитик-агента и
> настоящую маршрутизацию. **Слой 2** — action-агента, очередь событий (Redpanda)
> и нотификатора (асинхронная цепочка). Слой 2 поднимается профилем
> docker-compose, поэтому помечен как опциональный.

## Обзор

AskOps — это оркестратор и агенты-специалисты вокруг общих данных:

- **Оркестратор** (LangGraph + FastAPI) — точка входа. Принимает запрос,
  узлом-**роутером** классифицирует интент и по протоколу **A2A** вызывает
  нужного специалиста. Маршрутизация настоящая: знание → RAG, статистика →
  аналитик, действие → action.
- **RAG-агент** (FastAPI + A2A-сервер) — «мозг знаний». По вопросу делает
  kNN-поиск в OpenSearch, собирает контекст из найденных кусков и просит LLM
  сформировать ответ строго по контексту.
- **Аналитик-агент** (FastAPI + A2A-сервер) — «мозг цифр». По вопросу про
  статистику генерирует SQL (NL→SQL через LLM), **безопасно** выполняет его
  по базе тикетов (только SELECT, read-only) и отдаёт короткую сводку.
- **Action-агент** *(Слой 2, опционально)* (FastAPI + A2A-сервер) — «агент, который
  делает». На запрос-действие создаёт тикет в той же базе (статус «создан») и
  публикует событие в очередь.
- **Агент-нотификатор** *(Слой 2, опционально)* — consumer очереди: ловит событие
  и «отправляет» уведомление (консоль + файл + REST `/notifications`).
- **OpenSearch** — поисковый сервер с включённым kNN (база знаний RAG).
- **SQLite** — синтетическая база тикетов (читает аналитик, пишет action).
- **Redpanda** *(Слой 2, опционально)* — Kafka-совместимый брокер, один топик
  `ticket-events`. Связывает action-агента и нотификатора асинхронно.

## Схема

```
                        ┌────────────────────────────────────────────────────┐
  Пользователь          │                       AskOps                        │
  (REST /ask или        │                              A2A    ┌────────┐ kNN  │
   Streamlit UI)        │   ┌──────────────┐  ┌──────► RAG-  ─┼─► OpenSearch  │
        │               │   │ Оркестратор  │  │      │ агент  │  │(knn-индекс) │
        ├──────────────►│   │  (LangGraph) │──┤route └────────┘               │
        │   вопрос      │   │  + роутер    │  │      ┌──────────┐ SELECT(ro)   │
        │◄──────────────│   └──────────────┘  └────► Аналитик- ─┼─► SQLite     │
            ответ       │                       A2A │  агент    │  (тикеты)    │
                        │                           └──────────┘               │
                        └────────────────────────────────────────────────────┘
                                          эмбеддинги (sentence-transformers, RAG)
                                          LLM (GigaChat/OpenRouter за shared/llm.py)
```

## Маршрутизация (узел `route`)

Роутер оркестратора (`agents/orchestrator/router.py`) выбирает специалиста по
интенту вопроса (три маршрута):

1. **LLM-классификация.** Модель получает вопрос и отвечает одним словом —
   `rag`, `analyst` или `action`.
2. **Запасное правило по ключевым словам.** Если LLM недоступен, ответил
   непонятно или работает в mock-режиме — решает простое правило в порядке:
   действие («создай», «оформи заявку», «заведи тикет»…) → `action`;
   статистика («сколько», «динамика», «за май»…) → `analyst`; иначе → `rag`.

Так маршрутизация остаётся предсказуемой даже без реального LLM, а в графе
LangGraph видно, как принимается решение (узел `route` → `conditional_edges`).
Узел `call_action` устойчив к отсутствию action-агента (Слой 2 опционален): если
сервис не поднят, он возвращает понятный отказ, а не роняет запрос.

## Взаимодействие по A2A

- **RAG-агент = A2A-сервер.** Публикует Agent Card (по стандартному пути
  `/.well-known/agent.json`) с описанием своих возможностей: «отвечаю на
  вопросы по корпоративной базе знаний». Принимает A2A-сообщения и возвращает
  ответ.
- **Оркестратор = A2A-клиент.** Получает Agent Card RAG-агента, формирует
  A2A-сообщение с вопросом и получает ответ. Узел графа `call_rag` инкапсулирует
  этот вызов.

Такое разделение означает: оркестратор не знает, *как* специалист ищет ответ — он
лишь вызывает агента по протоколу. **Аналитик-агент** добавлен ровно по этой
схеме: своя Agent Card («отвечаю на вопросы по статистике тикетов»), тот же
A2A-транспорт (`ask_agent` в клиенте оркестратора), а роутер выбирает между ним и
RAG.

## Аналитик-агент и безопасность NL→SQL

Аналитик отвечает на вопросы по статистике обращений: «сколько инцидентов по
доступам за май», «среднее время решения по приоритетам». Конвейер:

1. **NL→SQL.** LLM получает описание схемы таблицы `tickets` и вопрос, генерирует
   один `SELECT`.
2. **Безопасное выполнение** (`agents/analyst/tickets_db.py`) — модели не доверяем:
   - коннект к SQLite открывается **read-only** (`mode=ro`) — изменения физически
     невозможны;
   - запрос проходит валидацию: ровно один оператор, начинается с `SELECT`/`WITH`,
     без запрещённых ключевых слов (`INSERT/UPDATE/DELETE/DROP/ATTACH/PRAGMA…`);
   - на выполнение вешается SQLite-**authorizer**, пропускающий только чтение;
   - результат ограничен лимитом строк.
3. **Сводка.** LLM превращает табличный результат в короткий текстовый ответ;
   сам выполненный SQL возвращается рядом — для прозрачности.

Любое нарушение на шаге 2 — понятный отказ пользователю, а не произвольный SQL от
модели. Это прямой ответ на пункт вакансии про безопасность агентов.

Данные тикетов **синтетические** (`agents/analyst/generate_tickets.py`, Faker,
фиксированное зерно): ~5000 обращений за год с правдоподобными перекосами
(доступов больше, чем поломок железа; пик в понедельник; редкие долгие инциденты).
Реального доступа к корпоративным системам у пет-проекта нет — это осознанная и
честно заявленная симуляция.

## Action-агент, очередь и нотификатор (Слой 2, опционально)

Второй тип межагентного общения — **асинхронный, событийный** — поверх синхронного
A2A. Поднимается профилем docker-compose (`--profile action`), чтобы не держать
Kafka постоянно.

Цепочка на запрос-действие («создай заявку на доступ к боевой БД»):

```
Оркестратор ──A2A──► Action-агент ──► SQLite (новый тикет, статус «создан»)
                          │
                          └──► Redpanda (топик ticket-events) ──► Нотификатор
                                                                      │
                                                       консоль + файл + /notifications
```

1. **Action-агент** (`agents/action/`) определяет категорию/приоритет заявки,
   пишет новый тикет (`REQ-…`) в общую базу тикетов и публикует событие
   `ticket_created` в топик `ticket-events`.
2. **Redpanda** — Kafka-совместимый брокер (один контейнер, без ZooKeeper),
   тот же протокол, что у Kafka.
3. **Нотификатор** (`agents/notifier/`) слушает топик и «отправляет» уведомление.

**Надёжность (важный пункт вакансии).** Запись тикета и публикация события
разделены: если брокер недоступен, `publish_event` возвращает `False`, тикет всё
равно создан, а пользователю честно сообщается, что событие не ушло. Цикл
нотификатора переживает падение брокера — ждёт и переподключается.

## Абстракции и подмена компонентов

- **LLM спрятан за `shared/llm.py`** (функция `ask_llm`). Весь код зовёт
  `ask_llm`, не зная про GigaChat. Подменить провайдера (на локальную модель или
  другой API) — поменять одну функцию.
- **Эмбеддинги — `shared/embeddings.py`** (локальная модель sentence-transformers,
  работает офлайн). Размерность вектора согласована с настройкой поля
  `knn_vector` в индексе OpenSearch.

## Поток данных при ответе

1. Пользователь → `POST /ask` оркестратора (или через Streamlit).
2. Оркестратор (LangGraph): узел `route` классифицирует интент →
   `conditional_edges` → `call_rag` / `call_analyst` / `call_action` (A2A-клиенты).
3a. **RAG-ветка:** эмбеддинг вопроса → kNN-запрос в OpenSearch → топ-N кусков →
   сборка контекста → `ask_llm(промпт с контекстом)` → ответ.
3b. **Аналитик-ветка:** NL→SQL → безопасное выполнение SELECT по базе тикетов →
   `ask_llm(сводка по результату)` → ответ.
3c. **Action-ветка** *(Слой 2):* создание тикета в SQLite → публикация события в
   Redpanda → ответ (нотификатор поймает событие асинхронно).
4. Ответ возвращается оркестратору по A2A → пользователю.

Офлайн-подготовка данных выполняется один раз при первом запуске, результат лежит
в именованных volume:
- **RAG:** `agents/rag/indexer.py` читает `docs/knowledge_base/`, бьёт на куски,
  считает эмбеддинги, кладёт в OpenSearch.
- **Аналитик:** `agents/analyst/generate_tickets.py` генерирует синтетическую базу
  тикетов в SQLite.
